 《OurMusic 音乐播放器项目开发文档》

一、项目概述

OurMusic是一款基于 Qt 6.8 + QML 开发的轻量级桌面音乐播放器。项目采用 QML 构建用户界面，C++ 实现核心数据与业务逻辑，通过 Qt 元对象系统实现前后端高效通信。主要功能涵盖本地音频文件导入、元数据解析、歌曲管理、播放控制、歌单（收藏夹）管理、播放列表队列控制等，旨在提供简洁、流畅的音乐管理体验。

团队成员：张文耀（组长）、李渝平（成员）、苟曦月（成员）



二、实际界面功能分析
当前实现的主界面布局如下（与初始设计存在差异，以代码为准）：

顶部标题栏：显示应用名称“OurMusic”和 Logo，无标签页切换，全局固定。
左侧歌单栏（MusicListTabBar）：以垂直列表展示所有歌单（集合），包括：
内置歌单：“全部音乐”（索引0）、“我喜欢的音乐”（索引1）
用户自定义歌单（通过“创建歌单”按钮添加）
每个歌单项可点击切换，选中项高亮（红色背景）
底部提供“创建歌单”和“导入歌曲”按钮
右侧主内容区（MusicListMenu）：展示当前选中歌单内的歌曲列表（SongItem 组件）。
每首歌曲显示：歌名、歌手、专辑、时长、喜欢状态
操作按钮：播放、喜欢、添加到播放列表、收藏（加入其他歌单）、删除（行为因歌单类型而异）
顶部显示歌单名称，并支持重命名和删除（仅自定义歌单）
提供“添加歌曲”按钮，从全局歌曲库中选取未加入当前歌单的歌曲
底部播放控制栏（PlayerControllerItem）：固定于窗口底部。
左侧显示当前播放歌曲名称
中间控制区：喜欢按钮、上一曲、播放/暂停、下一曲、循环模式切换（列表循环/单曲循环/随机）
进度条及时间显示（当前/总时长），支持拖拽跳转
右侧：收藏按钮（加入歌单）、音量控制（弹出垂直滑块）、播放列表按钮（显示当前播放队列对话框）

整体布局清晰，未使用 TabBar，而是采用左侧歌单列表 + 右侧内容区的结构，更符合音乐管理类应用习惯。



三、核心功能模块实现（基于代码）

模块一：歌曲数据管理（Song / SongBroker / SongFactory）

Song 类（`song.h/cpp`）：表示单首歌曲，包含元数据属性（名称、歌手、专辑、URL、时长、是否喜欢），提供 Q_PROPERTY 供 QML 绑定。
SongFactory 类（`songfactory.h/cpp`）：负责从本地文件 URL 构建 Song 对象。使用 `QMediaPlayer` 加载文件并解析元数据（标题、作者、专辑、时长），处理缺失元数据情况（返回“Unknown”占位）。
SongBroker 类（`songbroker.h/cpp`）：单例，管理全局所有歌曲对象（QList<Song*>）。提供增删查改接口，并支持持久化到 `songdata.json`（存储 URL 和 liked 状态）。启动时加载 JSON 并自动构建 Song 对象。

模块二：歌单管理（MusicCollection / CollectionBroker）

MusicCollection 类（`musiccollection.h/cpp`）：表示一个歌单（收藏夹），包含名称和歌曲 URL 列表。提供增删歌曲、查询等方法。
CollectionBroker 类（`collectionbroker.h/cpp`）：单例，管理所有歌单集合（QList<MusicCollection*>）。内置两个固定歌单（“全部音乐”、“我喜欢的音乐”），用户可通过“创建歌单”生成自定义歌单。持久化到 `collectiondata.json`（仅存储自定义歌单，内置歌单动态生成）。支持重命名、删除自定义歌单，以及动态刷新“全部音乐”和“我喜欢的音乐”内容。

模块三：播放队列控制（PlayList）

PlayList 类（`playlist.h/cpp`）：单例，管理当前播放队列（歌曲名+URL 列表）。支持添加、删除、清空、设置队列。维护当前播放索引，并提供 `nextIndex()` / `previousIndex()` 方法，根据播放模式（列表循环、单曲循环、随机）计算下一首索引。其状态（当前索引、播放模式）通过信号通知 QML 更新。

模块四：播放引擎与界面控制器（PlayerControllerItem）

PlayerControllerItem（`PlayerControllerItem.qml`）是 QML 中的核心播放控制组件，内嵌 `MediaPlayer` 和 `AudioOutput`。
功能：
  播放/暂停/停止
  上一曲/下一曲（调用 PlayList 索引计算）
  进度条与时间同步（通过 `onPositionChanged` 实时更新，使用 `utiles.js` 中的 `trans()` 格式化时间）
  音量控制（通过弹出滑块）
  喜欢当前歌曲（调用 `SongBroker.setSongLiked` 并刷新界面）
  收藏当前歌曲（触发歌单选择对话框，将歌曲 URL 添加到目标歌单）
  播放列表对话框（显示当前队列，支持点击切换和删除）

 模块五：UI 组件体系

 MusicListTabBar：歌单列表容器，支持滚动，底部有创建歌单和导入歌曲按钮。导入歌曲调用 `SongBroker.addSong` 并自动刷新。
 MusicListMenu：当前歌单的内容展示区。包含歌单名称、编辑/删除/添加按钮，以及歌曲列表（Repeater 嵌套 SongItem）。提供添加歌曲对话框（从全局未加入的歌曲中选择）、重命名对话框、收藏选择对话框。
 SongItem：单首歌曲的展示条目，包含播放按钮、信息文本、操作按钮（喜欢、添加到播放队列、收藏到歌单、删除）。删除行为根据当前歌单类型（全部音乐→彻底删除；我喜欢→取消喜欢；自定义→仅从该歌单移除）由 `Main.qml` 中的 `handleDeleteSong` 统一处理。
 Main.qml：主窗口，组合上述组件，负责初始化、连接信号槽、刷新歌单内容、处理删除逻辑、持久化保存等。

 模块六：数据持久化

 使用 JSON 格式存储两个文件：
   `songdata.json`：存储所有歌曲的 URL 和 liked 状态（便于恢复）。
   `collectiondata.json`：存储所有自定义歌单的名称及其包含的歌曲 URL。
 在程序退出或关键操作（增删改）后调用 `save()` 方法保证数据不丢失。



 四、类设计与关系（基于代码）
本项目的类设计围绕歌曲、歌单、播放队列和播放控制展开，各核心类及其协作关系如下：

Song 类
职责：作为歌曲实体，封装了歌曲的元数据（名称、歌手、专辑、URL、时长）以及喜欢状态。通过 Q_PROPERTY 暴露属性，便于 QML 直接绑定。
关联：被 SongBroker 聚合管理，是全局歌曲数据的基本单元。

SongFactory 类
职责：负责从本地文件 URL 解析音频元数据并构造 Song 对象。内部使用 QMediaPlayer 加载文件，通过事件循环等待元数据就绪（超时 3 秒），对缺失字段填充“Unknown”占位。
关联：仅被 SongBroker 使用，作为创建 Song 的工厂。

SongBroker 类（单例）
职责：全局唯一的歌曲容器，持有所有 Song 对象的列表。提供增、删、查、改（特别地，修改 liked 状态）接口，并负责将歌曲 URL 和 liked 状态持久化到 songdata.json。启动时从 JSON 恢复数据，并自动调用 SongFactory 构建 Song 对象。
关联：聚合 Song；被 CollectionBroker 和 QML 界面直接调用（通过 Q_INVOKABLE 方法）。

MusicCollection 类
职责：表示一个歌单（即歌曲集合），存储歌单名称和歌曲 URL 列表。提供添加、删除、查询歌曲 URL 的方法，以及清空操作。
关联：被 CollectionBroker 聚合，是歌单管理的基本单元。

CollectionBroker 类（单例）
职责：全局唯一的歌单容器，管理所有 MusicCollection 对象。内置两个固定歌单（索引0为“全部音乐”，索引1为“我喜欢的音乐”），并允许用户创建自定义歌单。提供按索引查找、删除歌单、创建新歌单等接口。持久化仅保存自定义歌单（内置歌单由程序动态生成），存储于 collectiondata.json。同时，它依赖 SongBroker 获取歌曲列表，以动态刷新“全部音乐”和“我喜欢的音乐”内容。
关联：聚合 MusicCollection；依赖 SongBroker 读取歌曲数据。

PlayList 类（单例）
职责：管理当前播放队列，内部维护一个歌曲信息列表（含名称和 URL）以及当前播放索引。提供添加、删除、清空队列、设置整个队列的方法。根据播放模式（列表循环、单曲循环、随机循环）计算下一首和上一首的索引，并通过信号通知界面更新状态。
关联：被 PlayerControllerItem（QML）直接调用，不依赖其他 C++ 类。

PlayerControllerItem（QML 组件）
职责：虽然位于 QML 层，但其内部嵌入了 QtMultimedia 的 MediaPlayer，作为实际的播放引擎。它调用 PlayList 获取队列信息，调用 SongBroker 更新喜欢状态，调用 CollectionBroker 执行收藏操作。它是连接 C++ 数据层和用户交互界面的核心控制器。

整体协作关系：
SongBroker 和 CollectionBroker 是数据层两大单例，分别管理歌曲和歌单，两者相互独立但通过 CollectionBroker 的刷新方法（reloadAllMusic、reloadLikedMusic）间接依赖。PlayList 独立管理播放队列，PlayerControllerItem 通过 QML 的全局导入方式直接调用这三个单例，实现播放、喜欢、收藏、队列控制等功能。类间通信采用 Qt 信号槽和属性绑定，耦合度较低。



 五、技术实现要点

元数据解析：使用 QMediaPlayer 的 metaData() 获取音频信息，并通过 QEventLoop 阻塞等待元数据加载完成（超时 3 秒），避免异步问题。QML 与 C++ 交互：通过 qmlRegisterSingleton 注册单例类（SongBroker、CollectionBroker、PlayList），在 QML 中直接调用其方法和访问属性。
时间格式化：utiles.js 提供 trans(duration) 函数，将毫秒转为 MM:SS 格式。
播放模式：PlayList 支持三种模式（0-列表循环，1-单曲循环，2-随机循环），由 nextIndex() 和 previousIndex() 计算。
UI 响应式：大部分尺寸使用相对布局（锚点、Layout），窗口缩放时元素自适应。
删除逻辑：在“全部音乐”中删除会从所有歌单移除并彻底删除歌曲文件引用；“我喜欢”中删除仅取消喜欢状态；自定义歌单中删除仅从该歌单移除。
收藏功能：通过“收藏”按钮弹出歌单选择对话框，将当前歌曲 URL 添加到目标自定义歌单（不包含内置歌单）。

 八、与原始开发文档的主要改动点

原始文档（《OurMusic 音乐播放器项目开发文档》第一版）基于设计构想编写，实际代码实现与原始文档存在以下差异，现以条目形式列出：

界面布局变化：原始文档计划顶部有三个标签页（推荐、播放器、我的），实际实现中取消了标签页，改用左侧歌单列表（MusicListTabBar）与右侧内容区（MusicListMenu）的组合，界面结构从“标签页切换”变为“侧边栏歌单导航”。

封面显示缺失：原始文档设计了专辑封面展示区域（含占位图），实际代码中 SongItem 和 PlayerControllerItem 均未实现封面渲染，仅展示文本信息（歌名、歌手等）。

播放列表入口不同：原始文档将播放列表按钮（三条横线）放在播放控制区，实际实现将该按钮置于底部控制栏右侧，点击后弹出对话框显示当前播放队列，而非内嵌界面。

推荐页面和我的页面被歌单系统替代：原计划中的“推荐”和“我的”页面在实际中被“全部音乐”和“我喜欢的音乐”两个内置歌单所覆盖，功能整合到同一主界面中，不再有独立页面。

用户系统被移除：原始设计包含 User 类和 FavoriteList 类，支持登录和个人收藏管理，实际代码未实现任何用户登录功能，收藏管理直接由 MusicCollection 和 CollectionBroker 全局管理，简化了权限控制。

PlayList 独立于播放器：原设计将 PlayList 作为 MusicPlayer 的组成部分，实际实现中将 PlayList 设为独立单例，PlayerControllerItem 通过 QML 调用其接口，类间耦合度降低。

元数据解析实现细节：虽仍使用 QMediaPlayer 的 metaData 接口，但实际增加了事件循环和超时机制（3秒），防止异步加载卡顿，增强了稳定性。

音量控制交互增强：原始文档仅提及音量滑块，实际实现将音量按钮置于底部控制栏，点击后弹出垂直滑块 Popup，且支持双击静音/恢复，交互更丰富。

播放模式细化：原计划仅提及“循环播放”，实际实现了三种模式（列表循环、单曲循环、随机循环），由 PlayList 的 playMode 属性控制。

删除操作根据歌单类型区分：原始文档未区分不同列表的删除行为，实际中“全部音乐”删除会彻底移除歌曲及所有引用，“我喜欢的音乐”删除仅取消喜欢，“自定义歌单”删除仅从该歌单移除，逻辑更符合用户预期。

新增“收藏到歌单”功能：原始文档只提到“收藏”即喜欢，实际增加了“收藏”按钮，可将当前歌曲加入任意自定义歌单（弹出选择对话框），扩展了收藏概念。

数据持久化具体方案确定：原始文档仅抽象提及“保存”，实际采用 JSON 格式，分别存储歌曲 liked 状态（songdata.json）和自定义歌单内容（collectiondata.json），明确了存储结构。
